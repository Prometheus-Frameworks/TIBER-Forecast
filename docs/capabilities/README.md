# Forecast governed capability paths

A Forecast **capability** is any signal, feature family, or model input that originates outside
`seasonalPprModel.ts`'s existing hand-picked feature set and is proposed for eventual production use.
`player-history` (see [`player-history-production-only-v0.md`](./player-history-production-only-v0.md))
is the first capability to complete this path end-to-end, and is the reference example every future
capability doc in this directory should follow.

## The path

Every governed capability -- player-history, and any future one (Rookies, TeamState, FORGE, or
anything else) -- must earn its way into a production Forecast path through the same sequence of
separately-reviewed, separately-decided steps. No step authorizes anything beyond opening the next
one; a positive decision at any stage is a ceiling, not a running total.

```text
capability identified
  -> owned source artifact           (a specific, governed, promoted upstream artifact -- never a
                                       candidate/unpromoted one -- identified by repo + path + sha256)
  -> mirror / rehearsal path         (Forecast-local mirrors refreshed from that artifact, with
                                       fail-closed source-identity verification; experimental/rehearsal
                                       modules that compute candidate features -- no production binding)
  -> validation                      (real-vs-baseline-vs-shuffled-control comparison, evaluated
                                       independently across at least two disjoint origins/seasons, no
                                       averaging across origins)
  -> threshold review                (the validated evidence is compared against a pre-registered
                                       acceptance framework; decides whether a production-binding
                                       *review* may even be opened -- does not bind anything)
  -> binding review                  (production Forecast paths, required artifact/report inputs, and
                                       outstanding prerequisite gates are inventoried; a real leakage
                                       audit is run; decides whether an *implementation* issue may be
                                       opened -- still binds nothing)
  -> implementation                  (the feature is actually wired, narrowest safe scope only, inert
                                       by default behind an explicit opt-in; a real scope/leakage audit
                                       confirms nothing outside the named consumer changed; requires
                                       human sign-off before any non-default activation)
  -> activation verification         (the merged implementation is independently re-exercised from the
                                       default branch -- both paths, fail-closed provenance, gate
                                       bypass attempts, determinism, full-repo scope -- before the
                                       capability is considered ready to activate)
```

## What every stage has in common

- **Narrowing, not escalating, permissions.** Each stage's decision enum is deliberately incapable of
  claiming more than "you may open the next stage's issue." None of them binds production, claims
  readiness, or authorizes a broader scope than what was actually reviewed at that stage.
- **Fail-closed identity everywhere.** Source artifacts are pinned by exact repo + path + sha256 +
  promotion reference, never trusted by convention or file path alone. A mismatch halts the pipeline;
  it never silently substitutes a "close enough" artifact.
- **`production_only` before anything richer.** A capability's narrowest, most defensible feature
  subset is the only one eligible for binding until a separately-reviewed, separately-cleared
  added-value bar justifies a richer feature set. Being validated is not the same as clearing that bar.
- **Inert by default.** An implemented capability does not change any existing caller's behavior until
  that caller explicitly opts in. Activation and implementation are different approvals.
- **Human sign-off is a distinct, non-automatable gate.** No automated review, however thorough, can
  satisfy the requirement that a human with authority over the affected production model explicitly
  signs off before real activation.
- **Evidence over assertion.** Every stage's report cites the exact prior documents/commits it
  depends on and is regenerated by re-running real code against real data, not hand-written.

## Adding a new capability doc

When a future capability (Rookies, TeamState, FORGE, or otherwise) completes this same path, add a
sibling file here named after its capability ID (e.g. `rookies-production-only-v0.md`) following the
structure of `player-history-production-only-v0.md`: capability name, source-of-truth artifact,
Forecast mirrors/reports, validation shape, and the decision emitted at each stage, ending with what
the capability does and does not authorize.
